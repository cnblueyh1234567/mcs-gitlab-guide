🔍 ts_rank คืออะไร — Full-text Search

📌 ทำงานยังไง?

PostgreSQL เก็บ column พิเศษชื่อ search_vector ไว้ใน ticket table — เป็นการแปลงข้อความใน ticket ให้เป็นรูปแบบที่ค้นหาได้เร็ว (tsvector)

1
เก็บ search_vector: เมื่อสร้าง ticket → PostgreSQL แปลง ticket_name + ticket_desc เป็น tsvector เก็บไว้ล่วงหน้า
2
แปลงคำค้นหา: ตอนหา related → แยกคำในชื่อ ticket ใหม่ออกมา แล้วแปลงเป็น tsquery
3
ts_rank คำนวณ: เปรียบ tsquery กับ search_vector ของทุก ticket → คืนค่า 0–1 ว่าตรงกันมากแค่ไหน
ตัวอย่าง:
ticket ใหม่: "ปัญหา Login ไม่ได้"
→ แยกคำ: ["ปัญหา", "Login", "ไม่ได้"]
→ สร้าง tsquery: 'ปัญหา' | 'Login' | 'ไม่ได้'

ticket เก่า A: "User Login ไม่ผ่าน" → search_vector มีคำ Login, ผ่าน → ts_rank = 0.75
ticket เก่า B: "แก้ UI หน้า Dashboard" → search_vector ไม่มีคำเลย → ts_rank ≈ 0
ข้อดี ts_rank: เร็วมาก เพราะ search_vector ถูก index ไว้แล้ว ไม่ต้องอ่านทุก row
ข้อเสีย: ดูแค่ "คำตรงกัน" — ถ้าใช้คำต่างกันแต่ความหมายเดียวกัน score จะต่ำ

🔤 similarity() คืออะไร — pg_trgm Trigram

📌 Trigram คืออะไร?

แบ่งข้อความออกเป็น กลุ่มละ 3 ตัวอักษร แล้วนับว่ากลุ่มที่เหมือนกันมีกี่กลุ่ม

ตัวอย่าง:
คำว่า "Login" แบ่งเป็น trigrams: "Log", "ogi", "gin"
คำว่า "Loginn" แบ่งเป็น trigrams: "Log", "ogi", "gin", "inn"

มี trigram เหมือนกัน 3 จาก 4 → similarity ≈ 0.75
1
ชื่อเหมือนกันมาก → similarity สูง (ใกล้ 1.0)
"Login ไม่ได้" vs "Login ไม่ผ่าน" → ~0.6
2
ชื่อต่างกันหมด → similarity ต่ำ (ใกล้ 0)
"Login ไม่ได้" vs "แก้ UI Dashboard" → ~0.05
3
Filter: similarity > 0.1 → ถ้าชื่อคล้ายกันน้อยกว่า 10% จะไม่เอาเข้า candidates เลย
ทำไมต้องมีทั้ง ts_rank และ trigram?
ts_rank ดี: ค้นหาข้อความในเนื้อหายาวๆ ได้เร็ว เช่น ค้นในคำอธิบาย ticket
trigram ดี: จับชื่อที่สะกดผิดนิดหน่อย หรือพิมพ์ต่างกันเล็กน้อยได้

เช่น "ชำระเงิน" vs "ชำระเงิน " (มีช่องว่างท้าย) → ts_rank อาจไม่จับ แต่ trigram จับได้
สรุปความต่าง:
ts_rank = ค้นหา เนื้อหา / คำค้น ในข้อความยาว (ดูทั้ง name + desc)
trigram = เปรียบ ชื่อ ticket ว่าตัวอักษรคล้ายกันแค่ไหน (ดูแค่ ticket_name)
tag_overlap_ratio คำนวณอย่างไร?
ticket ใหม่มี 4 tags → ticket เก่า match 2 ตัว → ratio = 2/4 = 0.5
จากนั้น 0.5 × 0.35 = 0.175

📊 Module Score — สุขภาพแต่ละ Module

stability  = 100 - (bugCount / total) × 200    ← bug น้อย = stability สูง
velocity   = (doneCount / total) × 100          ← งานเสร็จกี่ %
quality    = 100 - (bugCount / total) × 300     ← หักหนักกว่า stability
complexity = (total / 20) × 100                 ← ticket เยอะ = complex (ยิ่งสูงยิ่งไม่ดี)
risk       = (bugCount / total) × 100
             + (คนดูแลน้อย ? +30 : 0)

module_score =
  stability  × 0.25
  + (100 - complexity) × 0.15
  + velocity × 0.25
  + quality  × 0.25
  + (100 - risk) × 0.10

ตัวอย่าง — Module IPD

20 tickets, bug 2 ใบ, done 15 ใบ, 3 คนดูแล

มิติคำนวณได้น้ำหนักคะแนน

velocity(15/20)×10075×0.2518.75
quality100 - (2/20)×30070×0.2517.5
complexity(20/20)×100 = 100 → (100-100)0×0.150.0
risk(2/20)×100 + 0 = 10 → (100-10)90×0.109.0
